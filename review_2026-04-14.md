Date: 2026-04-14
Reviewer: Claude (Opus 4.6, 1M)
Scope: critical paths only (security, trading, runner, CLI)
Test baseline: `pytest tests/ -q` → 218 passed before and after inline fixes.

## Executive Summary

**Overall verdict: YELLOW — one blocker fixed inline, two residual HIGH issues that warrant a follow-up patch before the first real-money session.**

Two CRIT-class bugs were discovered and fixed inline during the review:

1. `ensure_clob_auth()` was passing the *bound method* `private_key_hex`
   (no parens) to `ClobClient`, not the hex string. This would have
   failed authentication at the first attempt to trade live — no real
   keys leaked, but live trading was impossible as shipped.
2. The `--confirm-real-trade` CLI flag on `pmbot run --live` was
   accepted but never propagated; the very first `place_order` call in
   the strategy loop would always raise `OrderSafetyError`. This is the
   same "live trading impossible" class, from a different angle.

Both are now fixed and tested (218/218). See inline fixes section below.

Everything else is YELLOW or green. No private-key leakage paths were
found. No silent bypass of `paper_mode` was found. Shutdown flow looks
correct. Keystore zeroization is in place. The risk guard, rate limiter,
and preflight are all straightforward.

The two residual HIGH findings are:

- A **hard-coded fake Safe proxy init-code hash** in `proxy_wallet.py`
  (H-01) which means the fallback CREATE2 derivation would compute an
  incorrect address. Upstream `py_clob_client.utilities.derive_proxy_address`
  is tried first so in practice we'd always take the upstream path — but
  the fallback, if ever taken, silently returns garbage.
- **Preflight live checks run with `paper_mode=False` short-lived trader**
  (H-02). No order is placed, but the explicit model_copy to paper_mode=False
  makes it slightly easier for a future refactor to accidentally take a
  live path. Low-risk today; worth defensive hardening.

**Go/no-go recommendation:** proceed to first live session only after:
- Verifying `py_clob_client.utilities.derive_proxy_address` is installed
  and used (the fallback in `proxy_wallet.py` is broken).
- Verifying `pmbot preflight --prompt-password` returns all green.
- Running `pmbot test-order --live --confirm-real-trade` with a single
  minimum-size order **before** launching `pmbot run --live --confirm-real-trade`.

## Findings by Severity

### CRIT (fixed inline)

#### C-01 — `private_key_hex` passed as method object, not value

- **File:** `src/pmbot/trading/trader.py:218`
- **Was:**
  ```python
  clob = ClobClient(
      ...
      key=self._ks.private_key_hex,   # BUG: passes bound method
      ...
  )
  ```
- **Impact:** ClobClient would either reject the "key" immediately or,
  worse, sign nothing — first live trade impossible.
- **Fix (applied):** `key=self._ks.private_key_hex()`. All tests pass.
- **Related:** preflight.py:189 already uses parens correctly; trader
  was the only site missing them.

#### C-02 — `--confirm-real-trade` never propagated on `pmbot run --live`

- **File:** `src/pmbot/main.py:141-178` + `src/pmbot/runner.py:846-855` +
  `src/pmbot/trading/trader.py` (no arm method)
- **Was:** `main.py` accepted `--confirm-real-trade`, logged "ARMED",
  but the flag was thrown away. The comment even said "we don't pre-flip
  _live_confirmed". Meanwhile `_dispatch_trading` in the strategy loop
  calls `trader.place_order(...)` without `confirm_real_trade=True`, so
  the first such call would raise `OrderSafetyError`.
- **Impact:** Live daemon could never post a first order — strategy
  loop would spam `OrderSafetyError` errors and log failures. Failure
  mode is conservative (no unwanted orders) but the advertised feature
  was non-functional.
- **Fix (applied):**
  - Added `PMTrader.arm_live_confirmed()` which sets `_live_confirmed=True`
    in live mode (no-op in paper).
  - `runner._main` accepts `live_confirmed` kwarg and calls
    `trader.arm_live_confirmed()` right after `ensure_clob_auth()`.
  - `main.py` passes `live_confirmed=confirm_real_trade` into
    `runner_run`.
  - Updated misleading comment.
- **Tests:** 218/218 still pass. The existing `test_first_live_confirm`
  still exercises the per-call path.

### HIGH

#### H-01 — Fake Safe proxy init-code hash in fallback derivation

- **File:** `src/pmbot/trading/proxy_wallet.py:55-57`
- **Issue:** `SAFE_PROXY_INIT_CODE_HASH = "0x3d2f8b31a3a1a1fd...0d"` is
  clearly a placeholder (note the `a1fd5bd9e35e...` suspicious pattern)
  and not the real keccak256 of the Safe 1.3.0 proxy creation code.
  Comment on line 109 even says "MVP … exact match requires PM's factory
  wrapper contract, so we emit a warning when we have to fall back".
- **Flow:** In `derive_safe_proxy_address`, the preferred path is
  `py_clob_client.utilities.derive_proxy_address(eoa)`. If that import
  or call fails, the fallback CREATE2 pipeline runs with the fake hash
  and returns a wrong address. `detect_proxy_wallet` then calls
  `eth_getCode` on the wrong address, gets `0x`, and reports
  `signature_type=0, funder=eoa` — i.e. silently misclassifies a
  Safe-proxy user as EOA.
- **Impact:** If the user has a Safe proxy *and* the upstream helper
  isn't available, the bot would attempt to sign as EOA and
  `set_api_creds` would fail at best, or post orders against the wrong
  funder at worst (rejected by CLOB).
- **Suggested fix:** Either
  (a) compute the real 1.3.0 proxy init-code hash at module import by
      running keccak256 on the known bytecode and inline the literal, or
  (b) remove the fallback entirely and raise a hard `UnlockError` if
      `py_clob_client.utilities.derive_proxy_address` is unavailable —
      fail-fast beats silent misclassification for real money.
- **Priority:** Fix before live. Low odds of hitting it (py-clob-client
  is a required dep per pyproject.toml) but the consequence is bad.
- **Mitigation today:** add to preflight: an explicit import check on
  `py_clob_client.utilities.derive_proxy_address`.

#### H-02 — Preflight balance check constructs a `paper_mode=False` trader

- **File:** `src/pmbot/trading/preflight.py:129-131`
- **Issue:** `trader = PMTrader(pm, ks_obj, cfg.model_copy(update={
      "trading": cfg.trading.model_copy(update={"paper_mode": False})
  }), ...)`. Preflight is advertised as "never signs, never places
  orders" but it flips paper_mode off to get live balance numbers. In
  the current code nothing in the call chain (`detect_proxy_wallet`,
  `get_balance_usdc`, `get_balance_matic`) would ever place an order, so
  this is safe. However:
  - A future refactor that calls a different method on `trader` from
    preflight would immediately be live — no additional gate.
  - The `session_factory or (lambda: None)` argument would break with a
    `TypeError` if any DB path is ever taken during preflight.
- **Suggested fix:** introduce a read-only `PMTrader.probe_balances()`
  that does not require flipping paper_mode off, or instantiate a
  narrower helper class that only exposes balance reads. At minimum,
  add a comment banner.
- **Priority:** MED today (no actual bug, defensive).

### MED

#### M-01 — `cancel_order` rate-bucket double charge possible

- **File:** `src/pmbot/trading/trader.py:551-561`
- `cancel_order` calls `await self._bucket.acquire(1)` *then* wraps the
  network call in `retry_async`. Every retry attempt after a transient
  failure does NOT re-acquire a token, but the server sees it. Total
  token consumption per logical cancel can be < attempts. Probably not
  an issue (bucket is 25/min, retry attempts small) but worth noting.

#### M-02 — `cancel_all_for_market` only takes bucket once for a multi-market list

- `cancel_orders(external_ids)` may be a batch RPC (1 token = many
  cancels). If the relayer counts them individually, the bucket
  under-counts. Confirm PM's rate-limit policy on batch cancels.

#### M-03 — `_dispatch_trading` catches broad `Exception` per plan

- File: `src/pmbot/runner.py:663-667`
- Exceptions including `RiskDeniedError` and `OrderSafetyError` are both
  caught with `logger.exception(...)` and treated identically as
  `errors += 1`. A `RiskDeniedError` is *expected* behaviour (budget
  cap hit); `OrderSafetyError` is a bug signal. Split them:
  - `RiskDeniedError` → `skipped`, log at INFO.
  - `OrderSafetyError` → `errors`, log at ERROR.
  - everything else → `errors`, log with traceback.
- Current behaviour is loud on risk-denied (alarming on-call) and
  indistinguishable from real safety-rail trips.

#### M-04 — `_mark_canceled` iterates via `in_(external_ids)` without batching

- File: `trader.py:656-673`. If the user has > a few thousand open
  orders (unlikely but possible after a bug), SQLite's `IN` clause has
  a max host parameter count. Batch if list > 500.

#### M-05 — `RiskGuard._load_level` trusts DB enum string; resets to OK silently on ValueError

- File: `risk_guard.py:91-94`. A corrupted row reports OK (permissive).
  Prefer to log a WARN and/or default to PAUSE_NEW. Actual probability
  of corruption: low.

### LOW

#### L-01 — `KeystoreError` masks the eth_account exception type in the fallback branch

- File: `keystore.py:84-85`. `except Exception as e: raise KeystoreError(f"keystore decryption failed: {type(e).__name__}") from e`.
  The `from e` preserves the chain for tracebacks (good). But the
  message only shows `type(e).__name__` — if eth_account raises an
  `InvalidPassword`-subclass, the user sees `keystore decryption failed:
  InvalidPassword` which is clear; if it raises
  `UnsupportedKDFError`, they get a less useful message. Consider
  including `str(e)` truncated to 200 chars — eth_account never
  embeds the password or the private key in exception messages
  (verified in the upstream source).

#### L-02 — `preflight.py` `repr(e)` in results

- Files: `preflight.py:101, 113, 173, 181, 197, 218`. `repr(e)` is fine
  for `httpx` / `web3` exceptions but `py_clob_client` has raised
  strings in the past that include request bodies; on the
  `L2 API creds derive` branch (line 197) an exception could in theory
  contain the signed message payload. The private key itself is not in
  that payload, but principle-of-least-info suggests catching and
  logging only `type(e).__name__`.

#### L-03 — `Keystore.unlock` logs the address at INFO

- File: `keystore.py:96`. Public address isn't a secret but emitting it
  to any log aggregator ties the operator identity to the deployment.
  Not a fix, just flagging.

#### L-04 — `runner._dispatch_trading` does not check `RiskGuard.halt_level()` before iterating plans

- File: `runner.py:631-668`. The `RiskGuard` check happens inside
  `PMTrader.place_order`. If the bot is in `HALT_ALL`, the loop still
  walks every plan, hits risk denial, logs exception, continues. CPU /
  log-noise only; no safety impact. Consider calling
  `trader._risk.check_daily_drawdown()` at the top of `_dispatch_trading`
  and short-circuiting if not OK.

#### L-05 — `planner._qty_from_capital` uses `round(max_qty, 2)`

- File: `planner.py:111`. Python banker's rounding + binary float
  representation can push the result slightly below `min_size` after the
  `max(qty, min_size)` line — the current `max` handles it, so this is
  fine. Flagging because a future edit that removes the max would
  re-introduce the bug. Add a unit-test comment.

#### L-06 — `rate_limit.TokenBucket.acquire` wait calculation uses stale `_refill` outside lock

- File: `rate_limit.py:53-55`. `wait_s = missing / self._refill` is
  computed inside the lock, then `asyncio.sleep` outside. Correct.
  Flagging because the `missing` value can be stale once sleep ends (new
  concurrent acquirers may have consumed more tokens) — the `while True`
  retry handles this. Behavior is correct; annotate for maintainers.

#### L-07 — `ORDER_PLACED`/`LIVE_ORDER_FAILURE` events don't record the `price`/`qty`

- File: `trader.py:493-504`. The `ORDER_PLACED` payload has
  `client_order_id`, `external_id`, `order_type` but not `price`/`qty`
  (they ARE in the `MyOrder` row). For forensic replay
  post-incident, having them in the event payload alongside
  `LIVE_ORDER_FAILURE` (which DOES include them) would help.
