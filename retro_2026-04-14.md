## What worked

1. **Background subagents were the force multiplier.** Each Phase was dispatched as an autonomous agent with a clearly-scoped prompt, ran ~30-90 min unattended, returned a summary. Main thread stayed interactive for user questions the whole time.

2. **paper_mode=true as the literal default.** Every switch in YAML, in code, and in CLI defaults to paper. Going live requires two independent flags (config + CLI confirm). Removed the single point of "oops".

3. **Test discipline held.** Every phase shipped with tests; we hit 218/218 at ship time. The /qa pass still surfaced 1 real bug (F-10) the unit tests didn't catch — integration scenarios matter.

4. **Real-data discovery outperformed paper reasoning.** Big turning points came from running the real pipeline:
   - First screen returned 0 selected → forced D-034's daily-R reframe.
   - First paper placement simulation showed 0/18 actually executable → forced D-046 single-side Tier C.
   - First balance check showed $0 allowance on EOA → discovered user is on PM Proxy (D-054), not the assumed EOA model.

5. **Decision log (DECISIONS.md) paid for itself.** By D-055 we had 54 decisions, each with rule + reason + how-to-apply. When the user came back hours later asking "why did we put 100k depth cap" we had an exact answer. The "Why:" line specifically kept me honest about what was user-specified vs. what was my inference.

6. **The doc sync loop (private → public repo) let the user's friend view progress in near-real-time** without exposing the full codebase. Simple shell script, big UX win.

---

## Where we burned time

1. **Network flakiness was the single biggest tax.** CLOB API through the user's shared-IP proxy throttled at ~40% base request success. Every agent hit it; we had to retrofit retries (D-036), circuit breakers, adaptive concurrency, multi-source σ fallback (D-038), fallback RPC (D-052). Maybe 30% of total engineering time went to network resilience.

2. **PM data model is not what docs suggest.** Three surprises:
   - Top-reward markets (NHL games) don't appear in Gamma `/markets?condition_ids=`; only via `/events?slug=` (D-040).
   - Browser-wallet users have a **Polymarket Proxy** via a custom factory, NOT a Gnosis Safe (D-054).
   - PM "Deposit" in the UI sends funds to the proxy, not allowance on EOA; bot reading EOA balance showed $4.90 after $5 deposit because $5 went to proxy.
   Each discovery cost 1-2 hours of wrong assumptions before surfacing in real data.

3. **Strategy aspiration vs. $100 reality.** Started with "Tier A long-dated σ ≤ 2%" which on real data was $0 selected because (a) σ undetectable without 7-day history, (b) candidate markets too thick, (c) min_size × price > cap for most. Multiple reframes (D-041, D-043, D-044, D-046) converged on "Tier C news single-sided" which fits $100 + allows 11 paper placements.

4. **False starts fixed in tests late.** Tick alignment (0.001-tick markets) wasn't in the initial test suite — showed up only during paper simulation. QA F-10 (paper bypasses risk gate) was invisible until integration scenario. Unit tests test what we remember to test; integration tests find what we forget.

## Where we got lucky

1. **User was patient and specific**. A less-engaged counterparty would have accepted the first-round "0 selected" screen as "broken bot" and bailed. The iterative dialogue was what let D-034 → D-054 emerge.

2. **The flaky proxy was intermittent, not always-down.** Had CLOB been fully unreachable, we'd have had to simulate with synthetic data only. The ~40% success rate was annoying but enough to find real bugs.

3. **`py-clob-client` covers ~95% of what we need.** If PM forced us to write EIP-712 signing by hand, this would have doubled the effort.

## Things to revisit post-live

1. **Review H-01**: `SAFE_PROXY_INIT_CODE_HASH` placeholder in `proxy_wallet.py`. Upstream path covers the common case, but the fallback misclassifies Safe users. Unlikely for browser-wallet users (who use PolymarketProxy, not Safe) but worth fixing.

2. **Daily relayer budget**: TokenBucket enforces per-minute; we lack a daily cap. 25 req/min × 1440 min = 36k/day which we won't hit, but a clean overage tracker would let us alert before approaching limits.

3. **Single-sided reward assumption unverified**. The Tier C strategy hinges on PM paying reward for single-sided LP at reward zone pricing. If they don't, we lose the value prop and must switch back to double-sided + upscale capital. 1 live test of $5 answers this.

4. **Monitor-to-trader cancel semantics**. Currently intraday cancel fires when a market is evicted from candidate_pool, but we don't cancel-then-replace when just the price moved — we cancel only at daily screen time. For LP做市 that's acceptable (time_in_zone > queue position per PM's 10k-sample-per-week reward model), but observability here is thin.

5. **Observability of own_share realized vs. estimated**. We estimate own_share at screen time using zone capital snapshot; actual fills drift that. No metric yet computes "my realized reward / my actual quoted share". Post-launch: add a daily reconciliation job.

6. **Monthly/weekly retro cadence**. This retro is the first one. Schedule: week 1 after live start, then monthly.

---

## What I would do differently

1. **Walk the PM deposit flow manually before writing the unlock pipeline.** Would have saved the D-054 proxy-discovery detour.

2. **Write a "happy-path simulation" smoke test earlier.** The build_plans → trader → my_orders loop wasn't exercised against real DB rows until Phase 3b paper runthrough, late. Had we fake-populated candidate_pool at Phase 2 and run a dry placement, the min_size × $1 affordability problem would have surfaced hours sooner.

3. **Be wary of "everything passed tests" being confused with "everything works"**. Unit tests hit 218/218 multiple times during phases where the integration would have returned 0/18 placeable orders. A tests-pass signal must not be mistaken for a system-works signal.

4. **Demand a double-check on assumed thresholds before locking them in.** σ ≤ 2% for Tier A was picked from the meeting transcript without verifying against actual PM market σ distributions. Every subsequent strategy struggle traced back to that.
